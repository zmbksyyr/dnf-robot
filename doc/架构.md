# dofrobot 当前架构与调度规则

生成时间：2026-06-23

本文档描述当前代码真实行为，用于代码审核、部署前检查和长测排障。文档不记录已经删除的 `.so hook`、`df_game_r patch`、旧喊话入口等历史路径。

## 1. 总原则

当前系统按三层职责收敛：

- 调度层决定：维护目标人数、策略状态、actor 容器、UID 租约、自动补人、缩容、坏 UID 回收和 cleanup。
- actor 层执行：一个 actor 是一个 UID 的运行容器，负责上线、移动、普通喊话、全服喊话、摆摊、下线。
- 基础能力落地：`RobotRuntime` 调用真实 Online、Move、Shout、Store、Logout 等动作；Create/Cleanup 属于结构性操作，由 manager 控制。

关键边界：

- Web/API 只提交命令，不绕过 actor 直接操作运行态基础能力。
- 运行态动作只对 actor 已接管的 UID 生效。
- 自动模式下，未接管 UID 的运行态命令返回 `uid_not_attached`。
- 手动模式下，未接管 UID 会先进入 `manual_attach`，创建必要 actor 并装入 UID，再执行命令。
- Logout 只让 UID 下线，不删除 UID，不清理数据库，不释放正常 UID 数据。
- Create/Cleanup/Delete 属于结构性操作，必须进入全局保护，避免和补人、租约、运行态动作交叉。

## 3. 主要模块

- `robot_auto_api.go`：自动系统生命周期。`StartAutoActions` 创建 supervisor；`StopAutoActions` 停止 supervisor；`SetAutoEnabled` 是 Web 自动开关。
- `robot_auto_scheduler.go`：supervisor 主循环。每秒 tick，先跑守卫，再维护目标、清理坏租约、回收异常 actor、分配 idle actor、更新指标。
- `scheduler_policy.go`：自适应策略。模式包括 `bootstrap`、`fill`、`stable`、`store_expand`、`pressure`、`breaker`、`maintenance`、`manual`。
- `robot_supervisor.go`：supervisor 对 actor、ledger、runtime 的协调。
- `robot_actor_registry.go`：Web/API 和调度接触 actor ownership 的边界。
- `robot_actor_ledger*.go`：actor slot、UID lease、blocked UID、扩缩容和快照。
- `robot_actor_api.go`：用户运行态命令入口，统一做 scheduler busy、uid attachment、manual attach 判断。
- `robot_actor_commands.go`：actor 命令执行，上线、移动、喊话、摆摊、下线都从这里进入 runtime。
- `robot_actor_tick.go`：自动行为 tick。只有自动开启、actor 为 auto 模式且 UID 可运行时，才自动移动、喊话、摆摊。
- `robot_cleanup.go`：cleanup 候选保护、停止 UID、批量删除假人相关数据库行。
- `robot_operations.go`：结构性操作保护和操作历史。
- `robot_world_shout.go`：全服喊话走 monitor `127.0.0.1:30303`，不依赖 `.so hook`。
- `robot_status.go`、`robot_auto_metrics.go`、`database_status.go`、`system_status.go`：状态聚合给 Web/API。

## 4. 自动调度流程

supervisor 每秒执行一次：

1. 读取当前行为配置。
2. 执行自动守卫。
3. 维护目标 actor 数。
4. 释放 broken lease。
5. cleanup blocked UID。
6. 回收不健康 actor。
7. 给 idle auto actor 分配 UID。
8. 更新 metrics 和 scheduler status。

自动守卫按优先级阻断：

- 自动关闭或配置 `auto_actions=false`：进入 `manual`，不补人、不分配 UID、不自动动作。
- Key 无效：停止自动 actor，状态进入 `maintenance`，原因 `key_invalid=...`。
- 存在 create/cleanup/delete 等结构性操作：进入 `maintenance`，暂停调度。
- 游戏端口不稳定：释放部分 auto actor，进入 `pressure`。
- breaker 生效：回收异常 actor，按批次释放 auto actor，进入 `breaker`。

## 5. 调度策略

策略不是固定配置，而是根据目标人数和运行状态动态调整：

- `bootstrap`：还没有 live snapshot。
- `fill`：`running + connecting < target`，且连接压力可接受，提升上线批次和启动速率。
- `stable`：running、connecting、store 状态处于稳定区间。
- `store_expand`：在线健康且摆摊空间充足，提高摆摊并发、概率和点位尝试次数。
- `pressure`：端口异常、连接堆积、pending actor 过多、CPU/内存/goroutine 压力过高时降速。
- `breaker`：熔断窗口内，降低上线和摆摊压力，并释放部分 actor。
- `maintenance`：结构性操作或关键前置条件异常。
- `manual`：自动关闭。

目标人数规则：

- `AutoTargetOnlineCount <= 0` 时默认目标为 `20`。
- `MaxOnlineRobots > 0` 时目标不超过该值。
- actor 扩容按 `SchedulerOnlineBatchSize` 限制。
- pending actor 超过阈值时暂停继续创建，避免 idle/connecting 堆积。
- 缩容优先停空 actor、异常 actor、登录中/断线 actor，再停摆摊 actor，最后停普通 running actor。

由调度推导的运行节奏：

- 自动移动间隔。
- 自动喊话间隔。
- 摆摊并发。
- 摆摊概率。
- 摆摊间隔。
- 摆摊持续时间。
- 摆摊 tick。
- 摆摊点位尝试次数。
- 摆摊失败冷却。
- 上线批次、上线启动速率、fill 超时。
- breaker 暂停时间、释放批次、端口异常释放批次。

## 6. 自动与手动

自动开启时：

- 调度维护目标人数。
- actor 数不足目标时按批次补 actor。
- idle actor 存在时调度继续装入 UID。
- actor tick 可以自动移动、喊话、摆摊。
- 用户只允许操作已经被 actor 接管的 UID。
- 用户操作未接管 UID 返回 `uid_not_attached`，不抢占自动调度。

自动关闭时：

- `SetAutoEnabled(false)` 写入配置并设置 `autoEnabled=false`。
- 进入 `actor_container=auto_stop`。
- 调用 `stopAutoActors(true)` 停止自动 actor。
- scheduler status 更新为 `manual`，reason 为 `auto_disabled`。
- supervisor loop 仍运行，但不自动补人、不自动分配 UID、不自动移动/喊话/摆摊。

手动模式下：

- 用户运行态命令可以触发 `manual_attach`。
- 调度创建必要 actor slot，把 UID 装入 actor，再由 actor 执行上线、移动、喊话、摆摊或下线。
- 手动 actor 仍由同一套 registry/ledger 管理；没有第二套手动运行路径。

## 7. 用户命令保护

所有托管运行态动作都经过 `prepareUserActorCommand`：

- 上线：`OnlineManaged`
- 移动：`MoveManaged`
- 普通喊话：`ShoutManaged(..., false)`
- 全服喊话：`ShoutManaged(..., true)`
- 普通+全服喊话：`ShoutBothManaged`
- 摆摊：`StoreManaged`
- 下线：`LogoutManaged`

命令拒绝规则：

- actor registry 不存在：返回 registry unavailable。
- create/cleanup 等结构性操作进行中：返回 `scheduler_busy`。
- actor 容器正在创建、销毁、装入、卸载：返回 `scheduler_busy`。
- 任意 actor 处于 `assigned`、`online`、`releasing`：返回 `scheduler_busy`。
- 自动模式下 actor 数未达到目标：返回 `scheduler_busy`，消息类似 `auto_filling actors=X target=Y`。
- 自动模式下还有 idle actor 等待装入：返回 `scheduler_busy`，消息类似 `auto_loading idle=N`。
- 自动模式下 UID 未接管：返回 `uid_not_attached`。
- 手动模式下 UID 未接管：进入 `manual_attach`，创建 slot、AttachUID，然后执行命令。

这个规则保证创建/销毁 actor、装入/卸载 UID、结构性删除期间，不接受运行态动作。

## 8. actor 规则

actor 是 UID 的唯一运行容器：

- actor 没有 UID 时不做运行态动作。
- `onlineDesired=false` 时，actor 保持离线目标。
- `onlineDesired=true` 时，actor 尝试确保 UID 在线。
- Move、普通喊话、全服喊话、摆摊执行前都会 `ensureOnlineForCommand`。
- 若 UID 未在线，actor 会先设置 `onlineDesired=true` 并等待上线确认。
- 超过 `OnlineConfirmTimeoutMS` 仍未 running，命令返回 `online_timeout`。
- Logout 调用 `logoutCurrentUID`，设置 `onlineDesired=false` 并清掉自动动作计划，但 UID 仍挂在 actor 上。
- actor stop/release 才释放 UID ownership。
- 自动移动、自动喊话、自动摆摊只在 `actor.mode=auto` 且 `autoActionsEnabled=true` 时运行。

## 9. UID 租约和坏 UID

UID ownership 由 ledger 管：

- `tryLeaseUID` 防止同一 UID 被多个 actor 同时持有。
- blocked UID 不允许再次 lease。
- actor 释放 UID 时删除租约。
- broken lease 会先阻断 UID，再由 cleanup 路径处理。
- cleanup 失败的 UID 保持 blocked，避免反复分配造成循环错误。

## 10. 创建、删除、清理

Create 和 Cleanup 都是结构性操作，不属于普通 actor 动作。

cleanup 保护规则：

- 默认 dry run 只列候选，不删除。
- `force=true` 才执行删除。
- 删除前先标记 cleanup pending。
- 若 registry 存在，先 `StopUIDs(uids, true)`；否则 fallback 到 Logout。
- 非内部 broken cleanup 会等待 5 秒，让下线和 actor 回收先完成。
- 删除前清掉自动摆摊 busy 状态。
- 批量删除在事务内执行。

候选保护规则：

- 只处理 `d_starsky.robot_registry` 中登记的假人，或指定范围内的孤立摆摊权限记录。
- `d_taiwan.accounts.accountname` 必须等于 UID 字符串，否则 protected。
- 非内部 confirmed broken cleanup 要求存在 `d_starsky.Dummylist` 行，否则 protected。
- registry account 必须等于 UID 字符串，否则 protected。
- 删除范围包含 UID 表、CID 表、billing cera/cera_point、摆摊权限和事件表；表不存在目标列时跳过该表。

这套规则用于避免清理假人时误删正常账号。

## 11. 摆摊规则

- 手动摆摊由用户命令进入 actor，再由 actor 调用 runtime Store。
- 自动摆摊由 actor tick 触发，前提是自动开启、actor 为 auto 模式、UID running、策略允许。
- 摆摊前会确保在线。
- 摆摊成功会按 scheduler 推导的持续时间设置 `storeUntil`，到期后 actor 自动执行过期处理。
- 摆摊过期、失败、手动 Logout、actor release 都会撤销临时摆摊权限并恢复 `Dummylist.function_type=0`。
- 摆摊失败不代表点位一定坏，物品、网络、服务波动都可能导致失败。
- 历史成功点位不会因为一次失败直接永久废弃。
- 点位策略目前优先中下区域，避开上方 NPC 密集区域。

## 12. 喊话规则

- 普通喊话走 actor/runtime 普通聊天路径。
- 全服喊话走 monitor megaphone TCP 包，地址固定为 `127.0.0.1:30303`。
- 全服喊话名字使用角色名，空名 fallback 为 `Robot`。
- 全服喊话消息最大 255 字节，名字最大 `0x1e` 字节。
- 旧 `.so hook` 喊话路径已经删除。
- 旧 `RobotManager.Shout` 双发入口已经删除，当前入口收敛为 managed/actor/runtime 路径。

## 13. Web 和状态卡片

- Web 只负责展示状态和提交命令。
- 页面按钮可用性依赖 Database、Key、Game Port、Monitor 等前置条件。
- 自动刷新只控制浏览器拉取状态频率，不应本地持久化 token。
- Web 页面显示文本尽量使用英文，避免编码问题。
- Web 诊断日志写在 `/root/robot_stdout.log`。
- 业务日志写在 `/root/config/log_robot`。

## 15. 重点审核点

- 自动关闭后是否稳定停在 `manual`，且不再自动补人。
- 手动模式下首次操作离线 UID 是否只创建必要 actor 并装入目标 UID。
- 自动开启期间容器创建/装载时，用户命令是否稳定返回 `scheduler_busy`。
- 自动模式稳定后，用户命令是否只允许作用于已接管 UID。
- cleanup 是否只删除假人 registry/Dummylist/accountname 能互相证明的 UID。
- 大批量删除时，列表状态、在线数、数据库数、actor ownership 是否最终一致。
- 长测时策略是否能在 `fill`、`stable`、`store_expand`、`pressure`、`breaker` 间合理切换。
