# dofrobot 当前架构与调度规则

生成时间：2026-06-20  
核对版本：4247ffe

本文档描述当前代码真实行为，不写未实现的设计。

## 1. 总原则

当前系统按三层职责收敛：

- 调度层决定：维护全局状态、策略、actor 容器、UID 装入/取出、自动补人、回收、清理。
- actor 层执行：一个 actor 是一个 UID 的运行容器，负责上线、移动、普通喊话、全服喊话、摆摊、下线。
- 基础能力落地：RobotRuntime 调用实际 Online、Move、Shout、Store、Logout、Cleanup、Create 等能力。

关键边界：

- Web/API 只提交命令，不直接操作运行态基础能力。
- UID 在 actor 内，运行态动作才有效。
- UID 不在 actor 内，自动模式下运行态命令会被拒绝；手动模式下可以由调度先创建 actor 并装入 UID。
- Logout 只是 actor 行为，只让 UID 下线，不释放 UID。
- 释放 UID、换 UID、删除 UID、清空 actor 属于调度/registry 的所有权操作。

## 3. 主要模块

- `robot_auto_api.go`：自动系统生命周期。`StartAutoActions` 创建 supervisor 并启动；`StopAutoActions` 停止 supervisor；`SetAutoEnabled` 是 Web 自动开关。
- `robot_auto_scheduler.go`：supervisor 主循环和自动调度守卫。
- `scheduler_policy.go`：策略推导，当前模式包括 `bootstrap`、`fill`、`stable`、`store_expand`、`pressure`、`breaker`、`maintenance`、`manual`。
- `robot_supervisor.go`：supervisor 对 actor、ledger、runtime 的协调。
- `robot_actor_registry.go`：Web/API 和调度接触 actor ownership 的边界。
- `robot_actor_ledger*.go`：actor 状态表、slot、UID lease、blocked UID、扩缩容和快照。
- `robot_actor_api.go`：用户运行态命令入口，统一做 scheduler busy、uid attachment、manual attach 判断。
- `robot_actor_commands.go`：actor 命令执行，上线、移动、喊话、摆摊、下线都在这里进入 runtime。
- `robot_actor_tick.go`：actor 自动行为 tick，自动移动、喊话、摆摊只在自动开启且 actor 为 auto 模式时运行。
- `robot_operations.go`：结构性操作保护和操作历史。
- `robot_status.go`、`robot_auto_metrics.go`、`database_status.go`、`system_status.go`：状态聚合给 Web/API。

## 4. 自动与手动

### 自动开启

Web 打开自动后，`SetAutoEnabled(true)` 写入配置并设置 `autoEnabled=true`。supervisor 不会重建，下一轮调度会自然从 `manual` 回到自动策略。

自动模式下：

- 调度维护目标人数。
- actor 数不足目标时补 actor。
- idle actor 存在时调度继续装入 UID。
- actor tick 可以自动执行移动、喊话、摆摊。
- 用户只允许操作已经被 actor 接管的 UID。
- 如果用户操作的 UID 未被 actor 接管，返回 `uid_not_attached`。

### 自动关闭

Web 关闭自动不是关闭 supervisor，而是切到手动：

- `SetAutoEnabled(false)` 写入配置并设置 `autoEnabled=false`。
- 进入 `actor_container=auto_stop`。
- 调用 `stopAutoActors(true)` 清理自动 actor。
- scheduler status 更新为 `manual`，reason 为 `auto_disabled`。
- supervisor loop 仍在运行，但 `handleAutoGuards` 只更新状态并返回，不再自动补人、自动分配 UID、自动移动/喊话/摆摊。

手动模式下：

- 用户命令可以触发调度创建 actor slot。
- 调度把 UID 装入 actor 后，再由 actor 执行上线、移动、喊话、摆摊或下线。
- 手动创建的 actor 仍由同一套 registry/ledger 管理，不另设一套手动 actor 管理器。

## 5. 用户命令规则

所有托管运行态动作都经过 `prepareUserActorCommand`：

- 上线：`OnlineManaged`
- 移动：`MoveManaged`
- 普通喊话和全服喊话：`ShoutManaged`、`ShoutBothManaged`
- 摆摊：`StoreManaged`
- 下线：`LogoutManaged`

命令拒绝规则：

- actor registry 不存在：返回 registry unavailable。
- create/cleanup 等结构性操作进行中：返回 `scheduler_busy`。
- actor 容器正在创建、销毁、装入、卸载：返回 `scheduler_busy`。
- 任意 actor 处于 `assigned`、`online`、`releasing`：返回 `scheduler_busy`。
- 自动模式下 actor 数未达到目标：返回 `scheduler_busy`，消息类似 `auto_filling actors=X target=Y`。
- 自动模式下还有 idle actor 等待装入：返回 `scheduler_busy`，消息类似 `auto_loading idle=N`。
- 自动模式下 UID 未被 actor 接管：返回 `uid_not_attached`。
- 手动模式下 UID 未被 actor 接管：调度进入 `actor_container=manual_attach`，创建 slot、AttachUID，然后再执行命令。

这个规则的目的很简单：创建/销毁 actor、装入/卸载 UID 时，不接用户运行态命令；稳定后才接。

## 6. actor 规则

actor 是 UID 的唯一运行容器。

- actor 没有 UID 时不做运行态动作。
- `onlineDesired=false` 时，actor 保持离线状态。
- `onlineDesired=true` 时，actor 会尝试确保 UID 在线。
- Move、普通喊话、全服喊话、摆摊在执行前都会调用 `ensureOnlineForCommand`。
- 如果 UID 未在线，actor 会先设置 `onlineDesired=true` 并等待上线确认。
- 超过 `OnlineConfirmTimeoutMS` 仍未 running，命令返回 `online_timeout`。
- Logout 调用 `logoutCurrentUID`，设置 `onlineDesired=false` 并清掉自动动作计划，但 UID 仍然挂在 actor 上。
- 自动移动、自动喊话、自动摆摊只在 `actor.mode=auto` 且 `autoActionsEnabled=true` 时运行。

因此，“下线”和“释放 UID”不是一件事：

- 下线：actor 执行 Logout，UID 还在 actor 内。
- 释放：调度/registry 从 actor 取出 UID。
- 删除：结构性操作，删除前需要先通过 registry 停止相关 UID。

## 7. 调度状态表

当前全局状态不是单一字段，而是多张状态表/快照组合：

- 数据库状态：角色/UID 是否存在。
- actorLedger：actor slot、UID lease、blocked UID。
- actor snapshot：actor 当前 UID、状态、busy、desired、运行态摘要。
- runtime status：游戏进程侧的 init/login/running/stop 等状态。
- operation state：当前是否存在 create/cleanup/delete 等结构性操作。
- actor container state：当前是否存在 `auto_stop`、`manual_attach` 等容器操作。
- scheduler status：当前策略、reason、压力、熔断、目标、速率等。
- system status：数据库、key、game port、game hook 等前置条件状态。

调度决策以这些快照为输入。Web 页面展示的是聚合后的结果，不是某一个字段的原始值。

## 8. 创建、删除、清理

- 创建角色属于 manager/调度层调用基础能力，不由 actor 自己创建。
- 删除角色属于结构性操作，应先停止相关 UID，再执行数据库/基础能力删除。
- cleanup 使用已有基础能力处理坏 UID，不新增一套删除实现。
- blocked UID 清理前要确认没有 actor 仍持有该 UID。
- 大批量删除期间，结构性操作会让调度进入保护状态，避免边删边补。

## 9. 摆摊规则

- 手动摆摊由用户命令进入 actor，再由 actor 调用 runtime Store。
- 自动摆摊由 actor tick 触发，前提是自动开启、actor 为 auto 模式、UID 已 running、策略允许。
- 摆摊成功会设置 `storeUntil`，到期后 actor 自动执行过期处理。
- 摆摊失败不代表点位一定坏，物品、网络、服务波动都可能导致失败。
- 历史成功点位不能因为一次失败直接降级为不可用点位。

## 10. Web 和状态卡片

- Web 只负责展示状态和提交命令。
- 页面按钮的可用性依赖数据库、key、game port、game hook 等前置条件。
- 自动刷新只控制浏览器拉取状态的频率，不应在本地持久化 token。
- Web 诊断日志写在 `/root/robot_stdout.log`。
- 业务日志写在 `/root/config/log_robot`。

## 11. 已防止的问题

- 自动补人超过目标：调度按目标容量计算，不应在现有数量已达目标后继续创建。
- 删除和补人交叉：结构性操作期间调度进入保护，不接运行态命令。
- 用户命令和 actor 装载交叉：`actor_container` 操作期间返回 `scheduler_busy`。
- login 卡住：actor 根据上线确认超时和健康状态暴露异常，调度再决定回收和换 UID。
- broken UID 未清理：ledger/cleanup 巡检确认 ownership 后调用基础清理能力。
- 下线误释放 UID：Logout 只改变在线期望，不释放 ownership。
- 全服喊话回归：当前基础能力验证版本已确认移动、摆摊、普通喊话、全服喊话正常。

## 12. 当前需要继续重点观察

- 自动关闭后，是否完全进入 `manual`，且不再自动补人。
- 手动模式下，首次操作离线 UID 是否只创建必要 actor 并装入目标 UID。
- 自动开启瞬间，容器创建/装载期间用户命令是否稳定返回 `scheduler_busy`。
- 自动模式稳定后，用户命令是否只允许作用于已接管 UID。
- 大批量删除时，列表状态、在线数、数据库数、actor ownership 是否最终一致。
- 600 长测时，策略是否能在 `fill`、`stable`、`store_expand`、`pressure`、`breaker` 间符合直觉地切换。
