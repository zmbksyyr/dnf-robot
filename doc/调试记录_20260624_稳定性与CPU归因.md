## CPU 归因

- 快照恢复和游戏服启动初期，CPU 高占用主要来自 `df_game_r cain01/cain05`，可到 `90%-100%`，此时 robot 主进程通常约 `3%-5%`。
- 600 长跑稳定阶段，CPU 排名常见为：
  - `df_game_r cain01` 约 `10%-15%`
  - `robot /root/robot` 约 `7%-11%`
  - `df_coserver_r` 约 `6%`
  - `df_game_r cain05` 约 `4%-8%`
  - `mysqld` 约 `2%-6%`
- robot 重启回填后发现一次 Web 子进程热点：
  - `/root/robot --web-admin ...` 曾达到约 `67% CPU`
  - 同时主 robot 进程约 `1%-2%`
  - 这说明高压期间 Web 状态页或 API 观察可能成为 CPU 热点，应作为后续优化项。
- 多次卡顿窗口表现为 `22/8111/8112/10011/30303` 全端口同时超时，不是单独 robot TCP API 阻塞。

## 未通过和风险

- 在 robot 重启后重新回填 600 的过程中，VM 进入长时间整机不可达状态：
  - SSH `22` 超时
  - robot API `8111` 超时
  - Web `8112` 超时
  - Game `10011` 超时
  - Monitor `30303` 超时
- `vmrun list` 仍能看到 VM 正在运行，说明宿主 VMware 管理接口存活，但 guest 网络/CPU 长时间不可响应。
- 结论：当前代码在 600 压力下可以收敛和恢复，但该 VM 对 600 + Web/API 观察 + 游戏服回填的组合压力仍可能整机卡死。后续应重点优化 Web/admin 状态查询 CPU 和大列表刷新开销，或在极限压测时降低 Web 自动刷新频率。
