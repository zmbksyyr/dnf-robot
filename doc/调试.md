# 调试与压测清单

本文定义调试方法、观察点和通过标准。目标是用尽量少的步骤覆盖更多行为，同时能看清当前架构里的关键状态：Web/API、数据库、Key、Game Port、Monitor、调度策略、actor 接管、UID 租约、基础动作、摆摊、结构性操作和异常保护。

## 2. 执行原则

- 能在同一轮压测里观察的项目合并观察，不重复开小场景。
- 每个阶段记录开始时间、结束时间、目标人数、日志偏移、Web 截图或关键数值快照。
- 不只看 `running`，必须同时看 `target / actors / leased / idle / blocked / broken / scheduler mode / reason / recent_operation`。
- 正常缩容、停止自动、手动下线只应 logout，不 cleanup 正常 UID。
- cleanup 只用于坏 UID、broken UID、明确选中 UID 或清理命令。
- Web 页面变红或刷新回登录页时，优先检查 `/root/robot_stdout.log`，确认是否有 `web admin exited`、`panic`、慢请求或 `auth rejected`。

## 4. 一轮烟雾覆盖基础能力

建议目标 `20`，运行 `10 分钟`。这一步覆盖 Web/API、配置热更新、基础动作、actor 接管、状态展示。

- [ ] Web 顶部 Robot / Game Port / Monitor / Database / Key / Auto / Online / Store 卡片正常。
- [ ] Database、Key、Game Port、Monitor 任一前置条件异常时，业务按钮置灰或命令被阻断；恢复后按钮可用。
- [ ] `autoStatus`、`schedulerStatus`、`systemStatus`、`databaseStatus`、`keypairStatus`、`robotsStatus count=10000` 正常返回。
- [ ] Web 列表显示所有假人，并能看到 actor 是否接管。
- [ ] `target=20` 后，actor 逐步接管 UID，`actors` 靠近目标，`leased<=actors`。
- [ ] 普通移动成功，Web/API 坐标可变化。
- [ ] 普通喊话成功，游戏内可见。
- [ ] 全服喊话成功，游戏内可见，日志包含 monitor/world shout 成功信息。
- [ ] 摆摊成功时 Web 显示 store 状态，游戏内可见摊位。
- [ ] 摆摊中的 actor 不被误判为空闲，不被重复分配 UID。
- [ ] 通过 Web API 或 robot TCP API 指定 UID 执行 Online / Move / Shout / World Shout / Store / Logout，命令投递到对应 actor。
- [ ] 自动模式下，未被 actor 接管的 UID 不执行运行态动作，返回 `uid_not_attached`。
- [ ] manual 模式下，调度稳定后 API 运行态命令可按需创建 actor、装载 UID，再由 actor 执行动作。
- [ ] 调度正在创建/销毁 actor、装载/卸载 UID、缩容/扩容、切换 Auto 或执行结构性操作时，用户运行态命令返回 `scheduler_busy`。
- [ ] 日志无持续 `panic`、`fatal`、`msg_queue_full`、`database is closed`。

## 5. 一轮升降压测覆盖调度

推荐一轮完成：`20 -> 100 -> 600 -> 20 -> autoStop`。

每个目标保持 `15 分钟`，每 `3 分钟`记录一次 Web/API/日志快照。若机器承载不足，可把 `600` 降到当前机器安全上限，但必须保留“升到高位再缩回”的过程。

### 观察点

- [ ] 扩容时 actor 数、租约数逐步靠近目标。
- [ ] 扩容优先使用小 UID，不重复租用同一 UID。
- [ ] 创建空间不明显超过目标；低库存或清空数据库时不创建到目标以上很多。
- [ ] `running` 可短暂低于目标，但 `actors / leased / scheduler reason` 必须能解释。
- [ ] `idle` 不长期大量堆积；pending actor 超过阈值时应进入 pressure 或降速。
- [ ] 缩容时优先释放空 actor、异常 actor、登录中 actor，再释放摆摊 actor 和普通 running actor。
- [ ] 缩容释放 UID 只 logout，不 cleanup 正常 UID。
- [ ] 缩容后多余 actor 不再移动、喊话、摆摊。
- [ ] `autoStop` 后自动 actor 停止，动作计数停止增长。
- [ ] `recent_operation` 与结构性操作状态一致，没有 create/cleanup 交叉执行造成状态不一致。
- [ ] CPU、内存、goroutine 不持续线性增长。
- [ ] Web 列表、卡片、scheduler 区在大批量操作期间仍能刷新，不长期空白。

### 动作覆盖

同一轮压测中同时观察：

- [ ] `move_success` 持续增长，`move_failed` 不持续增长。
- [ ] `shout_local_success` 持续增长，日志包含 `channel=local type=3`。
- [ ] `shout_world_success` 持续增长，日志包含 `channel=world type=80` 或 monitor megaphone 成功。
- [ ] 普通喊话和全服喊话互相独立，不串频道。
- [ ] `store_running/store_concurrent pN%` 能解释当前摆摊投放状态。
- [ ] 摆摊成功、失败重试、到期、取消都能在日志或 Web 状态中观察到。
- [ ] 摆摊到期、失败、Logout 或缩容释放后，小毛驴/摆摊权限被撤销，角色恢复普通状态。
- [ ] 历史成功点位不能因为一次物品失败或网络波动被永久降级为不可用点位。

## 6. 压测窗口注入故障

优先在 `100` 或 `600` 阶段执行，不单独开小场景。每次只破坏少量对象，便于追踪。

### 6.1 破坏已租 UID

- [ ] 记录 1 到 3 个已被 actor 接管的 UID。
- [ ] 删除或破坏其核心角色行，使其成为 broken UID。
- [ ] 调度巡检发现 `broken_lease` 或 missing core。
- [ ] Web/API 显示 broken 或 missing core，不把它算作正常在线。
- [ ] actor 先 release/logout 当前 UID。
- [ ] 调度调用已有 cleanup 基础能力，不新增独立清理路径。
- [ ] cleanup 成功后出现 `broken_cleanup_done`，失败则出现 `broken_cleanup_failed` 或 `broken_cleanup_skipped`。
- [ ] cleanup 失败的 UID 保持 blocked，不再被分配。
- [ ] actor 数不因坏 UID 处理而膨胀，调度能补位到目标。

### 6.2 Key 异常与恢复

- [ ] 临时移走或破坏一个 game key 文件。
- [ ] Key 卡变异常，业务按钮置灰或业务命令返回 key 错误。
- [ ] 调度停止新增 actor 和 UID 分配。
- [ ] 日志出现 key blocked 相关原因。
- [ ] 恢复 key 后，Key 卡正常，调度继续。

### 6.3 游戏端口异常与恢复

- [ ] 临时关闭游戏服，或把 `RobotGamePort` 指向未监听端口。
- [ ] Game Port 卡显示 closed。
- [ ] 调度不制造上线风暴，不继续创建/分配新 UID。
- [ ] 恢复游戏端口后，等待稳定窗口，调度继续补位。
- [ ] 恢复期间无大量 `online_failed` 风暴。

### 6.4 Monitor 异常与恢复

- [ ] 临时停止或阻断 `127.0.0.1:30303`。
- [ ] 全服喊话失败应明确报 monitor 连接或发送错误。
- [ ] 普通移动、上线、普通喊话、摆摊不应被 monitor 异常阻断。
- [ ] 恢复 monitor 后，全服喊话恢复。

### 6.5 大批量 cleanup

- [ ] 在高人数阶段选中数百个 UID 执行 cleanup。
- [ ] `recent_operation` 显示结构性操作进行中。
- [ ] 调度进入 maintenance 或等效保护，不边删边补到超目标。
- [ ] Cleanup 前相关 UID 先被 StopUIDs/logout。
- [ ] accountname 不等于 UID 字符串的记录被 protected，不删除。
- [ ] registry account 不等于 UID 字符串的记录被 protected，不删除。
- [ ] 非内部 cleanup 缺少 Dummylist 行的记录被 protected，不删除。
- [ ] 删除完成后 actor ownership、租约、running、目标人数重新一致。
- [ ] 页面卡片与列表最终一致，不长期停留在旧 running 数。

### 6.6 Web 子进程异常

- [ ] 压测中观察 Web 卡片是否突然全红。
- [ ] 若刷新回登录页，立即检查 `/root/robot_stdout.log`。
- [ ] 查找 `web admin exited pid=... err=...; restarting`，确认 web 子进程是否退出。
- [ ] 查找 `[WebAdmin] panic ...`，确认是否是某个 path 打崩 handler。
- [ ] 查找 `[WebAdmin] request ... duration=...`，确认是否是慢请求/阻塞。
- [ ] 查找 `[WebAdmin] auth rejected ... path=...`，确认是否是 token 失效后的旧页面自动刷新。
- [ ] 不允许通过本地持久化 token 绕过问题；session 只保存在 web 子进程内存中。

### 6.7 robot 崩溃或重启

- [ ] 压测中 kill robot。
- [ ] 重启后 Web/API 恢复。
- [ ] actor 状态表重新建立。
- [ ] 对已有 active UID 的接管不重复上线、不重复租约。
- [ ] 调度继续向目标收敛。

## 7. API 操作专项

这一项可穿插在 `20` 或 `100` 阶段，不另起一轮。逻辑验证以调用 Web API 或 robot TCP API 为准，不依赖人工点击页面。客户端只用于最终抽样确认少量基础能力的实际表现。

### 7.1 API 调用原则

- [ ] 使用 Web 的 `/api/call` 或 robot TCP API 调用命令，记录请求 payload、响应、日志偏移和 scheduler 快照。
- [ ] Web 页面只作为状态观察入口，不把“手点按钮成功”作为逻辑验证标准。
- [ ] 每个 API 调用后检查 `schedulerStatus`、`robotsStatus`、`autoStatus` 和 `/root/config/log_robot`，确认状态变化可解释。
- [ ] 用户客户端只最终观看个别 UID 的实际效果：移动、摆摊、普通喊话、全服喊话。
- [ ] 客户端确认只做抽样，不承担判断调度、租约、cleanup、blocked、pending 是否正确的职责。

### 7.2 API 覆盖命令

- [ ] 调用 `robotsOnlineAsync`，分别覆盖 `{count:N}` 和 `{uids:[...]}`。
- [ ] 调用 `robotsMove`，确认 API 返回成功后，再抽样看客户端坐标变化。
- [ ] 调用 `robotsShout`，确认普通喊话 API 和日志成功后，再抽样看客户端可见。
- [ ] 调用全服喊话相关 API，确认 monitor/world shout 日志成功后，再抽样看客户端可见。
- [ ] 调用 `robotsStoreAsync`，确认 API、日志和 Web store 状态成功后，再抽样看客户端摊位可见。
- [ ] 调用 `robotsLogoutAsync`，确认只 logout，不 cleanup 正常 UID。
- [ ] 调用 `cleanupRobotsAsync`，只用于明确选中坏 UID 或 cleanup 专项，确认 protected 规则生效。

### 7.3 API 场景验证

- [ ] Auto 关闭后，通过 API 确认策略显示 `manual`，调度先 logout 并清空原自动 actor。
- [ ] Auto 关闭且调度稳定后，通过 API 调用运行态动作；未接管 UID 应先被 actor 接管，再执行动作。
- [ ] Auto 打开后，现有 actor 计入目标容量；例如目标 200、已有 2 个 actor 时，只补 198 个。
- [ ] Auto 打开后，通过 API 操作未接管 UID 时，不允许抢占自动调度，应返回 `uid_not_attached` 或等效拒绝。
- [ ] 同一 UID 连续调用多个动作 API 时，actor 队列有序，不互相打断导致状态错乱。
- [ ] Store 进行中通过 API 调用 Logout 或触发缩容，确认能取消并回收权限/摊位状态。
- [ ] Cleanup 正在运行时，通过 API 调用 Online/Create，应被结构性操作保护阻断或排到安全路径，不交叉冲突。

## 8. 长时间破坏场景专项

目标是用尽量少的长时间测试覆盖复杂调度行为。不要只跑几分钟看正常路径，必须制造卡 login、大量 pending、持续失败、端口抖动、DB/状态不一致等场景，观察调度在错误长期累积后的表现。

推荐至少跑 `1 小时`，条件允许时跑 `2 小时`。优先在 `target=600` 极限人数下执行；如果 VM CPU 长期满载导致无法操作，可降到 `300`，但必须记录降级原因。

### 8.1 极限长跑基线

- [ ] 设置 `target=600`，开启 Auto。
- [ ] 保持至少 `60 分钟`，每 `5 分钟`记录一次快照。
- [ ] 不做人工干预时，观察 running、login/init、pending actor、failures、blocked、broken 是否持续累积。
- [ ] 观察调度是否能解释短时低于目标：`scheduler mode/reason`、`online_start_rate`、`online_batch_size`、pressure/breaker。
- [ ] 观察 Web/API 是否还能刷新，不能只看 robot 进程是否还活着。
- [ ] 观察 `/root/config/log_robot` 是否持续出现同类错误，例如 `online_failed`、`online_confirm_timeout`、`recycle_uid`、`broken_cleanup_failed`。
- [ ] 观察 VM CPU 卡顿时，系统是否只是降速/等待，而不是误判为网络断开后制造上线风暴。

### 8.2 制造卡 login / 大量 pending

在同一轮 `target=600` 长跑中执行，尽量一次制造足够多 pending，避免多个小场景看不出调度行为。

- [ ] 临时让一批登录连接停留在 `init/login` 或无法确认 `running`。可选方式：降低游戏端处理能力、临时限速/阻塞游戏端口、制造高 CPU 卡顿、让 `10011` 短时抖动。
- [ ] 至少制造 `50+` 个 pending/login/init；极限场景可制造 `100+`。
- [ ] 观察 `online_confirm_timeout` 是否按次数记录失败，而不是一次超时就 cleanup UID。
- [ ] 观察失败次数未达到阈值的 UID 是否能继续重试。
- [ ] 观察达到 `scheduler.bad_failures` 的 UID 才进入坏 UID 处理。
- [ ] 观察 pending 很多时调度是否降速、进入 pressure 或给出明确 reason。
- [ ] 观察 pending 堆积期间是否仍继续无限创建 actor 或重复租用同一 UID。
- [ ] 观察恢复游戏端后，pending 是否能逐步收敛到 running 或失败计数路径。
- [ ] 观察长时间 pending 后，Web 列表、scheduler 状态、actor 状态是否一致。

### 8.3 长时间错误累积

这一步重点看“错误累积 1 小时以后系统是否还能解释和收敛”。

- [ ] 在长跑中持续保留一部分坏 UID、pending UID、断线 UID，不要立刻 cleanup。
- [ ] 每 `10 分钟`记录一次失败次数分布：0、1、2、达到阈值。
- [ ] 观察 blocked/broken 是否只增长到合理范围，不应无限污染可用 UID。
- [ ] 观察 actor 是否会被坏 UID 长期占住，导致目标在线数无法补位。
- [ ] 观察 release、logout、recycle、cleanup 是否交叉运行导致状态不一致。
- [ ] 观察 `recent_operation` 是否长时间停留在旧操作。
- [ ] 观察调度是否出现重复创建、重复上线、重复 cleanup 同一 UID。
- [ ] 观察 CPU、内存、goroutine 是否因为错误堆积持续线性增长。

### 8.4 破坏场景组合测试

用一轮长跑组合覆盖，不要拆成很多短测试。

- [ ] `target=600` 跑稳后，制造 `50+` pending。
- [ ] pending 未完全恢复时，临时关闭 monitor，确认只影响全服喊话。
- [ ] monitor 未恢复时，短时关闭或抖动 game port，确认调度进入保护，不制造上线风暴。
- [ ] game port 恢复后，等待稳定窗口，确认调度继续补位。
- [ ] 再破坏 1 到 3 个已租 UID 的核心角色行，确认 broken 处理不影响其他 pending 收敛。
- [ ] 最后执行一次缩容 `600 -> 100`，确认 release/logout 不 cleanup 正常 UID。
- [ ] 缩容完成后再升回 `600`，确认失败历史不会导致大量正常 UID 被跳过或删除。

### 8.5 观察结论必须回答的问题

每轮长时间破坏场景结束后，记录结论：

- [ ] 卡 login 最多累计多少 pending？持续多久后开始收敛？
- [ ] 是否出现一次 login 超时就删除 UID？
- [ ] 失败次数达到阈值前，UID 是否有重试机会？
- [ ] 达到阈值后，坏 UID 是否被隔离/blocked/cleanup，且不再反复租用？
- [ ] 调度是否因为 pending 很多而继续无限扩容？
- [ ] 恢复 game port 后，系统是否能回到目标在线数？
- [ ] 1 小时以上错误累积后，Web/API、日志、scheduler reason 是否仍可解释？
- [ ] 是否出现需要人工判断但日志无法解释的状态？

## 9. 每 3 分钟记录的快照

- Web：Robot CPU、内存、goroutine、Database、Key、Game Port、Monitor、Auto、Online/Target、Store。
- Scheduler：mode、reason、online_start_rate、online_batch_size、move interval、shout interval、store interval/duration/concurrent、scale_up/down、breaker、port guard、recent_operation。
- Actor：idle、assigned、online、running、busy、releasing、actor_attached。
- UID：target、leased、blocked、broken、missing_core、数据库总数。
- Runtime：running、login/init、disconnect reason、store display ack。
- Pending/失败：pending/login/init 数量、`online_confirm_timeout` 次数、failure=1/2/阈值数量、blocked/broken 数量。
- 动作计数：move、local shout、world shout、store success、store failed、store expired、online success、online failed。
- 业务日志关键字：`panic`、`fatal`、`msg_queue_full`、`stop_timeout`、`online_confirm_timeout`、`broken_lease`、`recycle_uid`、`broken_cleanup_done`、`broken_cleanup_failed`。
- Web 诊断关键字：`WebAdmin`、`web admin exited`、`request pid`、`auth rejected`。

## 10. 通过标准

- Web/API 始终可用；短暂慢可以接受，长期空白不可接受。
- Database、Key、Game Port、Monitor 任一前置条件异常时，调度和按钮能阻断；恢复后能继续。
- actor 是 UID 的唯一运行容器；运行态动作只通过 actor 执行。
- Auto 关闭后调度进入 `manual`，不自动补人；用户运行态命令按需创建 actor。
- Auto 打开后退出 `manual`，已有 actor 计入目标容量，调度只补缺口。
- 调度容器不稳定时，用户运行态命令统一拒绝为 `scheduler_busy`，不排队、不抢 actor。
- 调度能动态扩容、缩容、补位、释放空闲，不是只按人数推导一次。
- 移动、喊话、摆摊节奏由 scheduler status 中的动态值解释，不依赖手工固定配置。
- 结构性操作不会与创建、删除、租约回收交叉到状态不一致。
- 目标人数不被明显突破；清空数据库或大批量删除后不会自动创建到目标以上很多。
- 大量 pending/login/init 时，调度必须降速或给出明确 reason，不允许无限扩容或无限重复租约。
- 卡 login 不能一次超时就 cleanup UID；必须先累计失败并给 UID 重试机会。
- 失败长期累积后，系统仍能解释状态：失败次数、blocked/broken、recent_operation、scheduler reason 不应互相矛盾。
- `target=600` 或降级高压目标运行 `1 小时` 后，CPU、内存、goroutine、错误计数不能持续线性失控。
- 坏 UID 能被巡检发现、释放、cleanup、blocked 或复用，不需要人工点新功能清理。
- 普通喊话、全服喊话、移动、摆摊都真实可见。
- 缩容和停止自动只 logout 正常 UID，不 cleanup 正常 UID。
- cleanup 不删除 accountname 不等于 UID 的正常账号。
- Web 子进程异常必须能从 `/root/robot_stdout.log` 追踪原因。
- 无持续 `msg_queue_full`、panic、fatal、CPU/内存/goroutine 失控。
