## 执行原则

- [ ] 能在同一次压测或恢复流程中观察的项目合并观察，不重复单独跑。
- [ ] 记录每个阶段开始/结束时间、目标人数、日志偏移、关键计数快照。
- [ ] `running` 只能作为状态参考，不能作为补 actor 或判断目标达成的唯一依据。
- [ ] 判断调度是否正确时，以 `target/actors/leased/idle/recycling/blocked_uids` 为主。

## 生命周期与 UID 回收

- [ ] actor 分配 UID 后进入上线流程。
- [ ] 上线成功后进入 running。
- [ ] `online_success` 增加。
- [ ] 上线失败后 `online_failed` 增加。
- [ ] 连续失败达到 `scheduler.bad_failures` 后标记回收。
- [ ] 超过 `bad_recover_sec` 后标记回收。
- [ ] 日志出现 `recycle_uid`。
- [ ] 回收前 actor release/logout 当前 UID。
- [ ] Supervisor 调用 cleanup。
- [ ] cleanup 成功后日志出现 `recycle_cleanup_done`。
- [ ] cleanup 失败后日志出现 `recycle_cleanup_failed`。
- [ ] cleanup 失败 UID 加入 blocked。
- [ ] blocked UID 本进程内不再分配。
- [ ] 回收完成后 actor 回到 idle。
- [ ] actor 重新分配新 UID。
- [ ] 回收期间 actor 总数不增加。

## 故障注入

故障注入优先在 100 或 600 压测窗口中执行，避免重复启动一轮。

- [ ] 压测中 kill robot。
- [ ] robot 重启后 actor 重新建立。
- [ ] robot 重启后能接管已有 active UID。
- [ ] 接管 active UID 时不重复上线。
- [ ] 压测中短暂关闭游戏端口。
- [ ] 游戏端口关闭期间不创建 actor、不分配 UID。
- [ ] 游戏端口恢复后调度继续。
- [ ] 压测中破坏一个 UID。
- [ ] 坏 UID 触发 release、cleanup、重新分配。
- [ ] 压测中删除 key。
- [ ] key 异常期间调度停止新增，业务命令阻断。
- [ ] 恢复 key 后调度继续。
