# 随机稳定性压测脚本

脚本：`tools/vm_random_stability.py`

用途：在 VM 内长期运行，自动执行 `doc/调试.md` 的核心调试流程，同时随机调整目标在线人数、少量 cleanup UID，并持续采集 CPU、调度状态和关键日志，用于观察假人系统一整晚的稳定性和 CPU 波动。

## 默认行为

- 默认场景：`full`。
- 运行时长：8 小时。
- CPU/状态采样：每 30 秒一次。
- 过滤日志快照：每 10 分钟一次。
- 随机目标人数：100 到 600。
- 随机调人间隔：20 到 40 分钟。
- 随机 cleanup 间隔：60 到 90 分钟。
- 每次 cleanup：1 到 3 个 UID。
- 单次运行最多 cleanup：30 个 UID。
- cleanup 优先选择异常、离线、未接管、missing core、suspect/disconnected 的 UID。
- 如果没有异常 UID，默认允许少量选择在线 UID，先 `robotsLogoutAsync`，等待 15 秒，再 `cleanupRobots force=true`。

## full 场景时间线

脚本启动后会先开启 auto，并在 8 小时内按时间执行以下固定调试动作，同时继续进行随机调人、cleanup、CPU/日志采样：

- `0 分钟`：设置 `target=20`。
- `2 分钟`：抽样执行 `robotsMove`、`robotsShoutLocal`、`robotsShoutWorld`、`robotsStoreAsync`。
- `10 分钟`：设置 `target=100`。
- `30 分钟`：设置 `target=600`。
- `90 分钟`：停止 monitor，验证全服喊话/状态影响，再执行 `/root/run` 恢复。
- `150 分钟`：执行 `/root/stop` 制造 game port down，等待后执行 `/root/run` 恢复。
- `240 分钟`：收缩到 `target=100`。
- `270 分钟`：回扩到 `target=600`。
- `330 分钟`：重启 `/root/robot`，再回到 `target=600`。
- `390 分钟`：再次确保 `target=600`。

如果只想跑纯随机调人/cleanup，不执行固定故障场景：

```bash
python /root/vm_random_stability.py --scenario random
```

## VM 上运行

```bash
python /root/vm_random_stability.py
```

推荐后台运行：

```bash
nohup python /root/vm_random_stability.py > /root/vm_random_stability.out 2>&1 &
```

查看输出目录：

```bash
ls -td /root/robot_stability_* | head -n 1
```

实时查看事件：

```bash
tail -f /root/robot_stability_*/events.log
```

查看采样 CSV：

```bash
tail -n 20 /root/robot_stability_*/samples.csv
```

## 常用参数

只跑 6 小时：

```bash
python /root/vm_random_stability.py --hours 6
```

不 cleanup，只随机调人数和采样：

```bash
python /root/vm_random_stability.py --no-cleanup
```

禁止删除在线 UID，只删除异常/离线候选：

```bash
python /root/vm_random_stability.py --no-allow-online-cleanup
```

降低压力到 100-400：

```bash
python /root/vm_random_stability.py --target-min 100 --target-max 400
```

指定输出目录：

```bash
python /root/vm_random_stability.py --out-dir /root/robot_stability_manual
```

## 输出文件

- `events.log`：人类可读事件和采样摘要。
- `samples.csv`：每 30 秒的结构化采样，包含 target、actors、leased、running、connecting、goroutines、load、top CPU、日志关键字计数等。
- `collected_logs.log`：每 10 分钟采集一次 `ps top`、`uptime`、`log_robot` 和 `robot_stdout.log` 的过滤日志。
- `summary.json`：脚本结束时的摘要。

## 判断重点

- `actors/leased/running/connecting` 是否能跟随随机目标收敛。
- `goroutines` 是否长期线性增长。
- `load1/load5/load15` 与 `top_cpu` 中 robot、web-admin、df_game_r、mysqld、vmtoolsd 的占比变化。
- 是否持续出现 `connect_queue_full`、`message_queue_full`、`timer_queue_overflow`、`panic`、`fatal`。
- cleanup 后是否出现 blocked/recycling 长期不回落。
